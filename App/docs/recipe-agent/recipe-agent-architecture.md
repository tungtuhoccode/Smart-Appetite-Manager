# Recipe Agent Architecture

Last updated: 2026-03-14

## Overview

The Recipe system is a multi-agent architecture with three agents working together, powered by the Spoonacular Food API. A router agent classifies user intent and delegates to specialized search agents, each equipped with Python tool functions that wrap Spoonacular endpoints.

```
User: "What can I cook?"
    ↓
OrchestratorAgent → routes to RecipeAssistant (via agent_card matching)
    ↓
RecipeAssistant (router) → classifies intent → delegates to:
    ├── RecipeInventorySearch  (pantry-based recipe search)
    └── RecipeGeneralSearch    (cuisine/diet/random exploration + ingredient intelligence)
```

## Agent Roles

| Agent | Role | Has Tools? |
|-------|------|-----------|
| **RecipeAssistant** | Router — classifies intent, gathers preferences, delegates | No (LLM-only) |
| **RecipeInventorySearch** | Finds recipes matching user's current pantry inventory | Yes (inventory + recipe tools) |
| **RecipeGeneralSearch** | Explores recipes by name/cuisine/diet, random inspiration, ingredient intelligence | Yes (recipe + ingredient tools) |

### Why Three Agents Instead of One?

A single agent with all tools would work, but splitting has advantages:

1. **Focused instructions** — Each agent's system prompt is tailored to its workflow, reducing confusion
2. **Tool isolation** — Inventory search needs the `get_ingredient_names` tool; general search doesn't
3. **Simpler debugging** — When recipe results are wrong, you know which agent to inspect
4. **Router pattern** — RecipeAssistant handles preference-gathering and disambiguation without cluttering the search agents

## How Agents Discover Their Tools

SAM agents understand their tools through **automatic introspection** of the Python function:

1. **Function name** → tool name the LLM calls
2. **Docstring** → tool description (tells the LLM *when* to use it)
3. **Type hints** (`str`, `int`, `bool`) → JSON Schema for parameters
4. **`Args:` in docstring** → description for each parameter
5. **Default values** → marks parameters as optional

The framework-injected parameters (`tool_context`, `tool_config`) are **hidden from the LLM** — the agent never tries to fill those in.

### Example: How the Agent Sees `complex_search`

The Python function:

```python
async def complex_search(
    query: str = "",
    ingredients: str = "",
    exclude_ingredients: str = "",
    diet: str = "",
    intolerances: str = "",
    cuisine: str = "",
    max_ready_time: int = 0,
    number: int = 5,
    tool_context: Any = None,       # hidden from LLM
) -> Dict[str, Any]:
    """
    Advanced recipe search with filters for diet, cuisine, intolerances, and prep time.
    Falls back to findByIngredients then random recipes if no results.
    """
```

What the agent's LLM receives (auto-generated by SAM):

```json
{
  "name": "complex_search",
  "description": "Advanced recipe search with filters for diet, cuisine, intolerances, and prep time. Falls back to findByIngredients then random recipes if no results.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query":              { "type": "string", "default": "" },
      "ingredients":        { "type": "string", "default": "" },
      "exclude_ingredients": { "type": "string", "default": "" },
      "diet":               { "type": "string", "default": "" },
      "intolerances":       { "type": "string", "default": "" },
      "cuisine":            { "type": "string", "default": "" },
      "max_ready_time":     { "type": "integer", "default": 0 },
      "number":             { "type": "integer", "default": 5 }
    }
  }
}
```

The LLM knows the parameter names and types, but **not** what valid values are (e.g., which diet strings Spoonacular accepts). That knowledge comes from the **agent instruction** in the YAML.

## Handling a Complex API: The Spoonacular Pattern

The Spoonacular Food API has dozens of endpoints, each with many parameters. Here's how we handle that complexity:

### Strategy: Split API Into Focused Tool Functions

Instead of exposing one giant function with 30+ parameters, we create **one function per use case**, each exposing only the relevant parameters:

| Tool Function | Spoonacular Endpoint | Purpose | Key Parameters |
|--------------|---------------------|---------|---------------|
| `complex_search` | `/recipes/complexSearch` | Advanced filtered search | query, ingredients, diet, cuisine, intolerances, max_ready_time |
| `get_top_3_meals` | `/recipes/findByIngredients` | Best matches for given ingredients | ingredients, number |
| `search_meals` | (delegates to above) | Simplified entry point | ingredient, category, area |
| `get_meal_details` | `/recipes/{id}/information` + `/analyzedInstructions` | Full recipe details | meal_id |
| `get_meal_details_bulk` | `/recipes/informationBulk` | Batch detail fetch | meal_ids (comma-separated) |
| `get_random_meal` | `/recipes/random` | Random inspiration | (none) |
| `parse_ingredients` | `/recipes/parseIngredients` | Structure free-text ingredients | ingredient_list, servings |
| `convert_amounts` | `/recipes/convert` | Unit conversions | ingredient_name, source_amount, source_unit, target_unit |
| `get_substitutes` | `/food/ingredients/substitutes` | Find ingredient swaps | ingredient_name |

### Why Split Instead of One Big Function?

| Approach | Pros | Cons |
|----------|------|------|
| One function, 30 params | Single tool to learn | LLM confused by too many params; passes irrelevant ones |
| One function per endpoint | Clear purpose, focused params | More tools to register in YAML |
| Grouped by use case (our approach) | LLM picks the right tool naturally; params match intent | Slight redundancy between tools |

### How the LLM Knows Valid Parameter Values

The function signature tells the LLM *what* parameters exist, but not *what values are valid*. For example, `diet: str` doesn't tell the LLM that Spoonacular accepts `"vegetarian"` but not `"veggie"`.

This is solved by putting domain knowledge in the **agent instruction**:

```yaml
# In recipe_assistant.yaml
instruction: |
  ...
  - Diet: gluten free, ketogenic, vegetarian, lacto-vegetarian, ovo-vegetarian,
    vegan, pescetarian, paleo, primal, low FODMAP, whole30
  - Cuisine: African, Asian, American, British, Chinese, French, Indian,
    Italian, Japanese, Korean, Mexican, Thai, Vietnamese, ...
  - Intolerances: dairy, egg, gluten, grain, peanut, seafood, sesame,
    shellfish, soy, sulfite, tree nut, wheat
  - Meal type: main course, side dish, dessert, appetizer, salad, ...
```

This pattern — **function signature for structure, agent instruction for domain knowledge** — is how you handle any complex API.

### Alternative Patterns for Complex APIs

If the Spoonacular API had even more parameters, here are other patterns that work:

#### Pattern A: Catch-All `extra_params` for Rarely-Used Options

```python
async def complex_search(
    query: str = "",
    ingredients: str = "",
    diet: str = "",
    number: int = 5,
    extra_params: str = "",       # Catch-all for advanced usage
    tool_context: Any = None,
) -> Dict[str, Any]:
    """
    Advanced recipe search with filters.

    Args:
        query: Search query (e.g., 'pasta with chicken')
        ingredients: Comma-separated ingredients to include
        diet: Diet filter (e.g., 'vegetarian', 'keto')
        number: Number of results (1-10)
        extra_params: Additional API params as key=value pairs separated by
            semicolons. Available: maxReadyTime (int), minCalories/maxCalories (int),
            intolerances (comma-sep), excludeIngredients (comma-sep),
            type (e.g., 'main course'), sort (e.g., 'popularity').
            Example: 'maxReadyTime=30;minCalories=200;sort=popularity'
    """
```

Use this when an endpoint has 20+ optional parameters and you want to keep the signature clean while still allowing power users (or a smart LLM) to access everything.

#### Pattern B: MCP Integration (No Python Code)

If Spoonacular had an MCP server, you could skip Python entirely:

```yaml
tools:
  - tool_type: mcp
    connection_params:
      type: streamable-http
      url: "https://mcp.spoonacular.com/mcp"
```

The MCP server would expose all endpoints as tools with full schemas. Use this when available — zero custom code needed.

## Tool Functions Reference

All recipe tools live in `src/recipe_agent/mealdb_tools.py`. They share common infrastructure:

- **API key**: Read from `SPOONACULAR_API_KEY` env var via `_get_api_key()`
- **HTTP client**: `_fetch_json()` (GET) and `_post_form()` (POST) using `urllib`
- **Error handling**: Custom `SpoonacularError` exception with HTTP status and response body
- **Async**: All tool functions are `async def`, blocking HTTP calls run via `asyncio.to_thread()`

### Search Tools

#### `complex_search`

The primary search tool. Wraps Spoonacular's `/recipes/complexSearch` with a fallback chain.

```
complex_search(query, ingredients, exclude_ingredients, diet, intolerances, cuisine, max_ready_time, number)
```

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `query` | str | `""` | Free-text search (e.g., "chicken pasta") |
| `ingredients` | str | `""` | Comma-separated ingredients to include |
| `exclude_ingredients` | str | `""` | Comma-separated ingredients to exclude |
| `diet` | str | `""` | Diet filter (vegetarian, vegan, keto, etc.) |
| `intolerances` | str | `""` | Comma-separated intolerances (gluten, dairy, etc.) |
| `cuisine` | str | `""` | Cuisine filter (italian, mexican, etc.) |
| `max_ready_time` | int | `0` | Max prep time in minutes (0 = no limit) |
| `number` | int | `5` | Number of results |

**Fallback chain**: complexSearch → findByIngredients → random recipes

**Returns**: Scored, normalized recipe list sorted by `final_score`.

#### `get_top_3_meals`

Quick ingredient-based search using `/recipes/findByIngredients` with scoring.

```
get_top_3_meals(ingredients, number=3, pantry_ingredients="")
```

#### `search_meals`

Simplified entry point that delegates to `complex_search` or `get_top_3_meals` based on whether filters are provided.

```
search_meals(ingredient, category, area, max_ready_time, intolerances, number)
```

#### `get_random_meal`

Fetches one random recipe from `/recipes/random`.

```
get_random_meal()
```

### Detail Tools

#### `get_meal_details`

Full recipe info + analyzed cooking steps for a single recipe.

```
get_meal_details(meal_id)
```

Calls two endpoints in parallel: `/recipes/{id}/information` and `/recipes/{id}/analyzedInstructions`.

#### `get_meal_details_bulk`

Batch fetch details for multiple recipes in one API call.

```
get_meal_details_bulk(meal_ids)  # e.g., "123,456,789"
```

### Ingredient Intelligence Tools

#### `parse_ingredients`

Parse free-text ingredient strings into structured data (name, amount, unit).

```
parse_ingredients(ingredient_list, servings=1)
```

`ingredient_list` is newline-separated text (e.g., `"2 cups flour\n1 tsp salt"`).

#### `convert_amounts`

Convert ingredient amounts between units (e.g., cups → grams).

```
convert_amounts(ingredient_name, source_amount, source_unit, target_unit)
```

#### `get_substitutes`

Find substitutes for an ingredient (e.g., "what can replace butter?").

```
get_substitutes(ingredient_name)
```

## Scoring Model

When recipes are returned from search, each gets a multi-factor score:

```
final_score = 0.45 × pantry_coverage + 0.25 × preference_match + 0.20 × prep_time + 0.10 × (1 - missing_penalty)
```

| Factor | Weight | Calculation |
|--------|--------|-------------|
| **Pantry coverage** | 45% | `used_ingredients / total_ingredients` |
| **Preference match** | 25% | Matches on diet, cuisine, intolerances |
| **Prep time** | 20% | 1.0 if under target time, degrades linearly |
| **Missing penalty** | 10% | `1.0 - min(missed / 10, 1.0)` |

Scoring is computed in `_compute_scores()` and attached to each recipe via `_normalize_recipe()`.

## Data Flow: End-to-End Example

### Inventory-Based Search: "What can I cook?"

```
1. User → OrchestratorAgent → RecipeAssistant
2. RecipeAssistant classifies: INVENTORY_RECIPE_SEARCH → delegates to RecipeInventorySearch
3. RecipeInventorySearch:
   a. Calls get_ingredient_names() → "chicken,rice,tomato,onion,garlic,soy sauce"
   b. Calls get_top_3_meals(ingredients="chicken,rice,...") → 3 recipe summaries
   c. Calls get_meal_details_bulk(meal_ids="123,456,789") → full details
   d. Returns conversational summary + ```recipe_data JSON block
4. RecipeAssistant relays response unchanged (preserving recipe_data block)
5. Frontend parses ```recipe_data → renders RecipeCard components
```

### Filtered Search: "Vegetarian Italian under 30 minutes"

```
1. User → OrchestratorAgent → RecipeAssistant
2. RecipeAssistant classifies: GENERAL_RECIPE_SEARCH
   - Extracts preferences: diet=vegetarian, cuisine=Italian, maxReadyTime=30
   - Delegates to RecipeGeneralSearch with preferences
3. RecipeGeneralSearch:
   a. Calls complex_search(diet="vegetarian", cuisine="Italian", max_ready_time=30)
   b. Returns scored results + ```recipe_data JSON block
4. RecipeAssistant relays response unchanged
5. Frontend renders recipe cards
```

### Ingredient Intelligence: "Can I swap butter for coconut oil?"

```
1. User → OrchestratorAgent → RecipeAssistant
2. RecipeAssistant classifies: INGREDIENT_INTELLIGENCE → delegates to RecipeGeneralSearch
3. RecipeGeneralSearch:
   a. Calls get_substitutes(ingredient_name="butter")
   b. Returns conversational response (no recipe_data needed)
4. RecipeAssistant relays response
```

## Frontend Integration

The recipe agents return a `recipe_data` JSON block at the end of every recipe response:

~~~
```recipe_data
[{"id": "123", "title": "Chicken Teriyaki", "image": "...", "scores": {...}, ...}]
```
~~~

The frontend parses this block and renders it using:

- **`RecipeCard.jsx`** — Card with image, match %, cooking time, diet badges, ingredients
- **`RecipeDetailsDialog.jsx`** — Full recipe modal with steps, nutrition, YouTube videos
- **`SavedRecipesGrid.jsx`** — Grid of saved/bookmarked recipes

Without the `recipe_data` block, the frontend falls back to plain text display.

## Adding a New Spoonacular Endpoint

To add a new Spoonacular API capability (e.g., meal planning, nutrition analysis):

1. **Add the tool function** in `src/recipe_agent/mealdb_tools.py`:
   ```python
   async def get_meal_plan(
       time_frame: str = "day",
       target_calories: int = 2000,
       diet: str = "",
       tool_context: Any = None,
   ) -> Dict[str, Any]:
       """
       Generate a meal plan for a day or week with target calories.

       Args:
           time_frame: 'day' or 'week'
           target_calories: Target daily calories (default 2000)
           diet: Diet filter (e.g., 'vegetarian', 'keto')
       """
       api_key = _get_api_key()
       params = {"apiKey": api_key, "timeFrame": time_frame, "targetCalories": target_calories}
       if diet:
           params["diet"] = diet
       url = f"{SPOONACULAR_BASE}/mealplanner/generate?{urllib.parse.urlencode(params)}"
       data = await _fetch_json(url)
       return {"status": "success", "plan": data}
   ```

2. **Register it** in the appropriate agent YAML:
   ```yaml
   tools:
     - tool_type: python
       component_base_path: "src"
       component_module: "recipe_agent.mealdb_tools"
       function_name: "get_meal_plan"
   ```

3. **Update the agent instruction** to tell the LLM when and how to use it:
   ```yaml
   instruction: |
     ...
     - If the user asks for a meal plan → use `get_meal_plan`.
       Valid time frames: "day", "week".
     ...
   ```

### Key Principles

- **Docstring = LLM description**: Write it for the LLM, not for developers. Explain *when* to use the tool.
- **`Args:` descriptions = parameter guidance**: Include valid values, formats, and examples.
- **Agent instruction = domain knowledge**: List valid enum values (diets, cuisines, etc.) that the function signature can't express.
- **One function per use case**: Don't make the LLM choose between 20 parameters — give it focused tools.
- **All params optional where possible**: Use sensible defaults so the LLM can call with minimal args.
